# Pet Asset Provenance Gate

`frontend/public/pet/` contains vendored browser runtime libraries and Live2D
model data. These files are not generated by the Rust backend and are therefore
separate from Cargo and npm dependency lockfiles.

## Current status

The checked-in inventory at [pet-asset-provenance.json](pet-asset-provenance.json)
is deliberately marked `unverified`. It is a complete inventory of the current
four library groups and seven model groups, but it is not a license grant or a
claim that redistribution is permitted.

The `integrity.sha256` values do bind the current on-disk files. They are
content baselines only: a matching digest does not upgrade an unverified source
or license to `verified`.

The completed source audit now has stronger attribution evidence, but it does
not close the release gate:

- `pixi.min.js` matches the PixiJS 6.5.10 MIT distribution after newline
  normalization, and `pixi-live2d-display.min.js` matches
  `pixi-live2d-display@0.4.0-beta.2`, also MIT. The vendored tree still lacks
  the required local LICENSE/NOTICE copies and preserved acquisition artifacts.
- `live2dcubismcore.min.js` matches Live2D Cubism SDK for Web 5-r.4. Cubism Core
  is proprietary; the repository does not contain the applicable vendor
  license, SDK acquisition record, or evidence authorizing redistribution by
  this release publisher.
- `live2d.min.js` was identified byte-for-byte as the historical Cubism 2.1.00_1
  runtime source. Live2D no longer provides that runtime download, and the audit
  could not obtain an authoritative historical license snapshot covering the
  vendored copy.
- Hiyori, Mao, Mark, Natori and Wanko, together with the Tororo/Hijiki source
  material, map to Live2D's official sample-data repository at immutable commit
  `ed1e0b714826d92469b9e51cacc3346f4e393f03`. The local Mao descriptor is a
  derived modification and Wanko is a subset, so exact local redistribution
  rights still require review.
- Live2D's current Natori model terms prohibit commercial use, modification and
  redistribution. The current `model-natori` payload is therefore a hard
  release blocker unless it is removed through an approved product decision or
  replaced with an asset carrying compatible rights.

The governing public references reviewed were Live2D's
[sample-model terms](https://www.live2d.com/en/learn/sample/model-terms/) and
[Free Material License Agreement](https://www.live2d.com/eula/live2d-free-material-license-agreement_en.html).
These findings are attribution and risk evidence only. All 11 groups remain
`unverified`; none may be promoted to `verified` without preserving the exact
source artifact, applicable license/notice, redistribution evidence and human
review required below.

## Required fields

Every direct item under `frontend/public/pet/lib/` and every direct model
directory under `frontend/public/pet/model/` must have exactly one `groups[]`
entry. A release-candidate entry must contain all of the following:

| Section | Required release-candidate fields |
| --- | --- |
| `provenance` | HTTPS `sourceUrl`, immutable `immutableRef`, downloaded-source `artifactSha256`, and `retrievedAt` timestamp |
| `license` | exact `identifier`, HTTPS `textUrl`, repository `noticePath`, and a URL or checked-in `redistributionEvidence` |
| `integrity` | `algorithm: "sha256-tree-v1"` and a current `sha256` tree digest |
| `review` | `status: "verified"`, `reviewedAt`, `reviewedBy`, and at least one URL or checked-in evidence item |

All SHA-256 values use lowercase `sha256:<64 hexadecimal characters>` format.
The source artifact hash identifies the upstream archive or package used for
the review. The integrity digest binds the exact locally vendored payload.

`sha256-tree-v1` recursively sorts paths using POSIX `/` separators, hashes
each file's bytes with SHA-256, then hashes the UTF-8 records
`relative-path`, NUL, lowercase file digest, newline. A single-file library
uses `.` as its relative path. Symlinks and unmanaged library/model-root files
are rejected so a verified group cannot silently point outside the reviewed
tree.

## Gate behavior

Normal developer builds and tests do not run this gate. The regular release
input check validates the inventory structure and reports its unverified state,
but does not treat that state as a build failure:

```powershell
node scripts/verify-release-inputs.mjs
```

To print the current `sha256-tree-v1` values after an approved asset change,
use `node scripts/verify-release-inputs.mjs --print-pet-asset-digests`. That
helper only reports local content; it never changes `review.status` or accepts
an asset for release.

Signing or release-candidate validation must use the strict mode from a clean
checkout:

```powershell
node scripts/verify-release-inputs.mjs --require-clean --release-candidate
```

`SYNTHCHAT_RELEASE_CANDIDATE=1` or `SYNTHCHAT_SIGNING=1` also enables strict
mode, and strict mode rejects a missing `--require-clean`. It fails when any
group remains unverified, required evidence is missing, the declared group set
does not exactly match the vendored tree, or the local tree digest does not
match its manifest value. The root `npm run verify:release-ready` command
executes this same strict invocation.

When a legitimate asset update is proposed, review the source and license
first, add the required notice/evidence files, update the corresponding entry,
then run the strict command before producing or signing a desktop package.
Do not remove the Pet UI or its assets merely to bypass this gate. If legal
review determines that an asset such as Natori cannot be shipped, an approved
product decision to remove or replace that asset is a compliance action and
must be reflected in the manifest and release evidence.
